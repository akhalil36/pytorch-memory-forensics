THE PREMISE

An attacker with a memory dump can steal the model.

Most work on model theft assumes query access to an API. This project takes the adversary's other path: physical memory. If a model is running, its full parameters and structure are sitting in RAM — and with the right offsets, they can be read straight out.

The pipeline runs end to end: train a custom CNN on CIFAR-10, export it to TorchScript, load it in a C++ inference binary compiled with full debug symbols, then pause execution at the forward pass and capture the machine's memory. From that single image, GDB and a custom Volatility 3 plugin reconstruct the model layer by layer.

train CNN → TorchScript .pt → C++ inference (debug build) → gdb breakpoint @ forward → LiME capture → Volatility 3 plugin → recovered model
STEP · THE NETWORK

A CNN built from scratch, trained on CIFAR-10.

No pretrained models. A three-block convolutional feature extractor feeds a two-layer classifier, trained on an 80 / 10 / 10 split.

model.py
class CNN(nn.Module):
    def __init__(self):
        super().__init__()
        self.features = nn.Sequential(
            nn.Conv2d(3, 32, 3, padding=1), nn.ReLU(), nn.MaxPool2d(2,2),
            nn.Conv2d(32, 64, 3, padding=1), nn.ReLU(), nn.MaxPool2d(2,2),
            nn.Conv2d(64, 128, 3, padding=1), nn.ReLU(), nn.MaxPool2d(2,2),
        )
        self.classifier = nn.Sequential(
            nn.Flatten(),
            nn.Linear(128 * 4 * 4, 256), nn.ReLU(),
            nn.Linear(256, 10),
        )

    def forward(self, x):
        x = self.features(x)
        x = self.classifier(x)
        return x

After training, the model is exported to TorchScript with torch.jit.script and loaded by a C++ program that calls module.forward(inputs). The CMake build sets CMAKE_BUILD_TYPE Debug so every symbol survives into the binary — which is what makes the later memory archaeology possible.

STEP · GDB @ 0x5555584273f0

Reading PyTorch's C++ internals by hand.

With a breakpoint set on the forward-pass line, the process is frozen. From here every object — the input tensor, the module, each layer's weights — is walked through raw memory using field offsets pulled straight from PyTorch's TensorImpl.h.

gdb info inferior reports the process ID as 5749 — the same PID Volatility recovers later from the memory image, the first confirmation the two views line up.

Decoding the input tensor

The inputs IValue vector and the raw input tensor share the same impl. Dumping 32 quadwords at the implementation address and mapping them against the struct layout reveals the tensor's shape and strides:

Shape [1, 3, 32, 32] — exactly one CIFAR-10 image. The pattern now repeats for every layer.

Walking the module's slots

Printing model._ivalue_.target_->slots_ yields four slots: a training flag, None, then self.features at 0x5555583ef4d0 and self.classifier at 0x5555584237b0. Descending into features exposes the individual layers; the first Conv2d sits at 0x5555583da440, with its weight tensor as slot 0 and bias as slot 1.

The layer's attribute strings — stride, padding, dilation, padding_mode — are recovered the same way, decoded byte-for-byte from the ClassType object. The Linear layers follow identically: linear_1.weight resolves to shape [256, 2048], and following its StorageImpl's DataPtr to 0x7fffe23ff040 lands on 2,097,152 bytes of trained float data — the correct size for a [256, 2048] tensor.

Key detail

The addresses had to be gathered twice. Moving between VMs relocated everything, so the entire GDB pass was repeated on the final x86_64 machine — the addresses on this page are the ones that match the captured image.

STEP · LiME + dwarf2json

Freezing RAM at the breakpoint.

With the process paused mid-forward-pass, a second shell loads the LiME kernel module and writes the entire system's memory to disk in a Volatility-readable format.

capture.sh
# process is paused at the forward-pass breakpoint in another shell
sudo insmod lime-*.ko path=$HOME/memory.lime format=lime

# build the kernel symbol table for this exact build
sudo ./dwarf2json linux \
  --elf /usr/lib/debug/boot/vmlinux-$(uname -r) \
  --system-map /boot/System.map-$(uname -r) \
  > ubuntu-$(uname -r).json

The symbol table has to match the kernel build, not just its version — a mismatched architecture banner makes Volatility fail silently, which turned into the project's single biggest time sink (see the pitfalls below).

STEP · VOLATILITY 3

Confirming the process, then the layout.

Standard plugins first establish ground truth: the process exists in the image, and its memory map matches what GDB saw. Then volshell verifies individual addresses byte-for-byte.

PID MATCH

linux.pslist finds process 5749 · infer in the image — the exact PID GDB reported before the capture.

linux.proc.Maps --pid 5749 lays out the process's virtual memory areas. The heap spans 0x555555585000 – 0x5555584a5000 — roughly 47 MB where most of the live model tensors reside — while libtorch_cpu.so occupies a ~1.4 GB range of executable code.

Byte-for-byte verification in volshell

After switching the current layer to process 5749, a small dump helper reads memory at the GDB addresses. Every value matches. The input tensor at 0x5555583edd70, the module at 0x5555584273f0, the conv2d weight tensor, the "conv2d" and "Linear" ClassType strings, the linear-layer float weights — all identical across the live debugger and the frozen image.

STEP · CUSTOM PLUGIN

Automating the whole recovery.

The final deliverable: a Volatility 3 plugin that takes a PID and reconstructs the model — weights, architecture, and forward-pass code — without a human walking pointers by hand.

  1. Resolve the process & heap

    _find_process matches the PID via pslist; _build_proc_layer sets up virtual-to-physical translation; _get_heap_ranges scopes the search to the heap VMA found earlier.

  2. Read each TensorImpl

    _read_tensor_impl unpacks ndim, sizes, and strides from the documented struct offsets, then follows the StorageImpl to the raw DataPtr.

  3. Extract the weights

    _extract_weights reads n_elems little-endian floats from the data pointer — the actual trained parameters, straight out of RAM.

  4. Recover architecture from strings

    _extract_architecture scans the heap for ClassType names (conv2d, Linear, BatchNorm, ReLU, MaxPool) and reads each layer's attribute strings.

  5. Recover the forward-pass code

    _extract_forward_code scans for the TorchScript operators the forward graph invokes — aten::conv2d, aten::relu_, aten::max_pool2d, aten::linear — and locates them in the image.

The forward graph itself, extracted from the compiled TorchScript, shows the exact execution order the plugin corroborates:

graph(%self.1 : __torch__.model.CNN, %x.1 : Tensor):
  %features.1  = prim::GetAttr[name="features"](%self.1)
  %x0.1     = prim::CallMethod[name="forward"](%features.1, %x.1)
  %classifier.1 = prim::GetAttr[name="classifier"](%self.1)
  %11       = prim::CallMethod[name="forward"](%classifier.1, %x0.1)
  return (%11)

THE HARD PART

Four ways to make Volatility fail silently.

The forensics only works when the memory image, the kernel symbols, and the tooling all agree on the same architecture and build. Getting there took several dead ends.

✕
GitHub Codespaces

The kernel headers weren't accessible from the container, so the banner string couldn't be read at all.

✕
Ubuntu 26.04 VM (UTM)

The banner was now readable, but mismatched — the Ubuntu release was too new for a clean symbol match.

✕
Ubuntu 24.04 VM (ARM64)

Same banner mismatch. Volatility 3's Linux plugins don't hold up on ARM64 for this kernel line — pslist returned nothing, kmsg page-faulted, kallsyms crashed.

✓
Emulated x86_64 VM (UTM)

Slow under emulation on Apple Silicon, but every standard plugin ran and every address matched. The GDB pass was redone here so the addresses lined up with the new capture.
